Реализуй desktop приложение, которое умеет работать с YANG моделями.
Приложение должно поддерживать создание, редактирование YANG моделей как с помощью графического интерфейса, так и с помощью текстового редактора.
По YANG модели должно строиться дерево узлов, при нажатии на каждый узел должно выводиться описание узла, его ограничения и родительские узлы.
Поддерживаться импорт и экспорт моделей.
Возможность с помощью графического интерфейса указываться основные характеристики переменной, такие как описание, условия, ограничения.
Интерфейс не должен быть перегруженным.
Приложение должно быть полностью кросс-платформенным с Linux.
Приложение должно поддерживать YANG 1.1 согласно RFC 7950.

Пользовательская инструкция по работе с приложением находится в `YangDesignerInstruction.md`.

## Предложения

1. Использовать JavaFX как основной UI-стек, так как проект уже создан на JavaFX и имеет Maven-сборку.
2. Разделить приложение на слои: модель YANG-документа, парсер/валидатор, сервисы импорта-экспорта и UI.
3. Не писать полноценный YANG-парсер вручную на первом этапе. Лучше подключить готовую библиотеку для разбора YANG или сначала реализовать ограниченный внутренний AST, а затем заменить его на промышленный парсер.
4. Сделать основной экран трехпанельным: слева дерево узлов, по центру текстовый редактор YANG, справа свойства выбранного узла.
5. Синхронизировать текстовый и графический режимы через единую модель документа, чтобы изменения из GUI сразу отражались в тексте и наоборот.
6. Добавить валидацию с понятными сообщениями: строка, узел, тип ошибки, рекомендация по исправлению.
7. Поддержать YANG 1.1 согласно RFC 7950. В MVP выделить минимальный набор конструкций: module, namespace, prefix, yang-version, import, container, list, leaf, leaf-list, type, description, mandatory, default, config, when, must.
8. Для кросс-платформенности собрать и проверить запуск отдельно на Windows и Linux. В дальнейшем можно добавить jpackage-дистрибутивы.

## План работ

### Этап 1. Подготовка основы проекта

1. Переименовать шаблонные классы `HelloApplication`, `HelloController` и `hello-view.fxml` в рабочие имена приложения.
2. Убрать неиспользуемые зависимости из `pom.xml` или обосновать их использование.
3. Создать базовую структуру пакетов: `app`, `model`, `parser`, `service`, `ui`, `validation`.
4. Добавить стартовый главный экран с меню, пустым деревом, текстовым редактором и панелью свойств.

### Этап 2. Модель данных YANG

1. Описать внутренние классы для YANG-документа и узлов дерева.
2. Поддержать свойства узлов: имя, тип, описание, ограничения, условия, родитель, дочерние узлы.
3. Реализовать преобразование модели в дерево для UI.
4. Покрыть модель и преобразование базовыми unit-тестами.

### Этап 3. Импорт и парсинг

1. Реализовать открытие `.yang` файлов.
2. Построить первичный парсер или подключить готовую библиотеку.
3. Отображать структуру файла в дереве узлов.
4. Показывать ошибки разбора без падения приложения.

### Этап 4. Текстовый редактор

1. Добавить редактор с моноширинным шрифтом, номерами строк и индикатором несохраненных изменений.
2. Реализовать сохранение, сохранение как и перезагрузку файла.
3. Синхронизировать дерево с текстом после успешного разбора.
4. Добавить базовую подсветку синтаксиса YANG.

### Этап 5. Графическое редактирование

1. Реализовать выбор узла в дереве и отображение его свойств.
2. Добавить формы редактирования описания, типа, условий и ограничений.
3. Добавить создание и удаление основных типов узлов.
4. Генерировать корректный YANG-текст после изменений через GUI.

### Этап 6. Валидация

1. Проверять обязательные поля модуля: module, namespace, prefix.
2. Проверять уникальность имен дочерних узлов в рамках родителя.
3. Проверять базовые типы и обязательные ограничения.
4. Выводить список ошибок и предупреждений с переходом к узлу или строке.

### Этап 7. Импорт, экспорт и упаковка

1. Поддержать импорт одного файла и набора связанных моделей.
2. Поддержать экспорт в `.yang`.
3. Добавить примеры YANG-моделей для ручной проверки.
4. Настроить сборку исполняемого дистрибутива для Windows и Linux.

### Этап 8. Полировка интерфейса

1. Упростить меню и оставить только основные действия: создать, открыть, сохранить, экспортировать, проверить.
2. Добавить горячие клавиши для частых действий.
3. Проверить читаемость интерфейса на разных разрешениях.
4. Добавить короткую документацию по запуску и сборке проекта.

## Ближайший MVP

MVP стоит ограничить следующим объемом:

1. Открытие одного `.yang` файла.
2. Отображение дерева `module/container/list/leaf`.
3. Просмотр описания, ограничений и родителей выбранного узла.
4. Редактирование текста и сохранение файла.
5. Минимальная валидация структуры.
6. Linux-сборка через текущий Maven-профиль.

## Текущее состояние

В проекте уже заложена первая рабочая основа:

1. Главный экран JavaFX с деревом YANG-узлов, текстовым редактором и панелью свойств.
2. Открытие и сохранение `.yang` файлов.
3. Минимальный разбор YANG 1.1-конструкций для MVP.
4. Просмотр и редактирование имени, типа, описания и ограничений выбранного узла.
5. Создание дочерних `container`, `list` и `leaf` через GUI.
6. Генерация YANG-текста из внутренней модели после изменений в GUI.
7. Базовая валидация обязательных полей модуля и типа `leaf`.
8. Unit-тесты для парсера, валидатора и генератора.
9. Statement AST для сохранения неизвестных и vendor-specific statements при обратной генерации текста.
10. RW/RO-индикаторы в дереве, встроенная справка `F1` и выпадающий выбор YANG-типа.
11. Номера строк в редакторе и переход к узлу по двойному клику на сообщение валидации.
12. Перезагрузка текущего файла, отдельный экспорт в `.yang`, очистка Linux-дистрибутива перед упаковкой.
13. Базовая подсветка синтаксиса YANG в редакторе на RichTextFX.
14. Переход к определению узла в редакторе при выборе узла в дереве.
15. Отдельные вкладки для редактирования текущего узла и создания нового дочернего узла.
16. Детерминированный порядок вставки substatements при генерации новых узлов.

## Текущая поддержка YANG 1.1 RFC 7950

На текущем этапе реализована MVP-поддержка, а не полный промышленный RFC 7950-парсер.

Уже поддерживается:

1. Разбор базовой структуры `module`, `submodule`, `import`, `include`, `revision`, `extension`, `feature`, `identity`, `typedef`, `container`, `list`, `leaf`, `leaf-list`, `choice`, `case`, `grouping`, `uses`, `augment`, `rpc`, `action`, `input`, `output`, `notification`, `deviation`, `deviate`, `anydata`, `anyxml`, `enum`, `bit`.
2. Чтение `yang-version`, `namespace`, `prefix`, `type`, `description`, `key`, `when`, `must`, `mandatory`, `default`, `config`, `presence`, `min-elements`, `max-elements`, `pattern`, `range`, `length`, `units`, `base`, `if-feature`, `status`, `reference`, `organization`, `contact`, `belongs-to`, `revision-date`, `value`, `position`.
3. Диагностика строк объявления узлов.
4. Проверка, что модуль использует `yang-version 1.1`.
5. Проверка обязательных `namespace` и `prefix`.
6. Проверка уникальности имен дочерних узлов внутри родителя.
7. Проверка наличия `type` у `leaf` и `leaf-list`.
8. Проверка `list key` и ссылки ключа на дочерний `leaf`.
9. Проверка конфликта `leaf-list default` и `min-elements`.

Следующие шаги по RFC 7950:

1. Расширить statement AST позициями символов и сохранением исходного форматирования.
2. Добавить строгую грамматику cardinality/order для statements.
3. Реализовать разрешение `import`/`include` между несколькими файлами.
4. Добавить полноценную валидацию `typedef`, `identity`, `feature`, `if-feature`, `deviation`, `augment`, `extension`.
5. Улучшить GUI-редактирование так, чтобы оно работало напрямую с выбранным statement.

## Запуск и сборка

Для сборки нужен JDK 21. Если `JAVA_HOME` не задан глобально, его можно указать перед запуском Maven Wrapper.

На Windows:

```powershell
$env:JAVA_HOME='C:\Program Files\Android\openjdk\jdk-21.0.8'
.\mvnw.cmd test
.\mvnw.cmd javafx:run
```

Обычная сборка:

```powershell
.\mvnw.cmd package
```

Linux-дистрибутив:

```powershell
.\mvnw.cmd -Plinux-dist package
```

Профиль `linux-dist` очищает папку дистрибутива перед копированием файлов, чтобы не сохранять устаревшие зависимости от предыдущих сборок.

Пример модели для ручной проверки находится в `examples/basic-system.yang`.
